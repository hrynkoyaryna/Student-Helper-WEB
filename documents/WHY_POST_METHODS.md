# Чому вибрано POST методи для RateLimiting?

## Обґрунтування

### 1. **POST методи змінюють дані**
- GET методи - тільки читають
- POST методи - створюють, оновлюють, видаляють
- POST потребує більшого захисту

### 2. **POST менш частовий за GET**
- Користувачі часто читають дані (GET)
- Користувачі рідко пишуть дані (POST)
- Логічно обмежувати менш частовий тип запиту

### 3. **POST більш уразливий до атак**
```
GET  /Tasks/Index              → Можна викликати багато разів (читання)
POST /Tasks/Create             → Спам завдань (DDoS на БД)
POST /Tasks/DeleteConfirmed/5  → Видалення даних (найкритичніше!)
```

### 4. **Вибір конкретних методів**

#### TasksController.DeleteConfirmed (POST) - ⚠️ Найкритичніше
```csharp
[RateLimitingFilter(maxRequests: 20, timeWindowSeconds: 60)]
public async Task<IActionResult> DeleteConfirmed(int id)
```
- Видалення необоротної операції
- Ліміт найменший (20 запитів)
- Найбільший ризик від зловживання

#### TasksController.Create (POST) - 🟡 Важливо
```csharp
[RateLimitingFilter(maxRequests: 40, timeWindowSeconds: 60)]
public async Task<IActionResult> Create(TaskCreateEditViewModel model)
```
- Створення нових записів
- Потребує захисту від спама
- Ліміт середній (40 запитів)

#### NotesController.Create (POST) - 🟡 Важливо
```csharp
[RateLimitingFilter(maxRequests: 30, timeWindowSeconds: 60)]
public async Task<IActionResult> Create(NoteCreateEditViewModel model)
```
- Створення нотаток
- Аналогічно завданням
- Ліміт дещо нижче (30 запитів)

## Рекомендації для інших методів

### GET методи (читання) - Менше потребуватимуть RateLimiting
```
Index()           - Частовий запит, ліміт >100 запитів
Details()         - Менш частовий, ліміт ~80 запитів
Search()          - Залежить від кількості користувачів
```

### POST методи (запис) - Потребуватимуть RateLimiting
```
Create()          - Важливо, ліміт 30-50 запитів
Edit() [POST]     - Важливо, ліміт 30-50 запитів
Delete()          - КРИТИЧНО, ліміт 10-30 запитів
```

### DELETE методи (видалення) - Найстрогіший RateLimiting
```
DeleteConfirmed() - КРИТИЧНО, ліміт 10-20 запитів
BulkDelete()      - КРИТИЧНО, ліміт 5-10 запитів
```

             ▼

## Коли вибирати RateLimiting?

| Сценарій | RateLimiting | Причина |
|----------|------------|---------|
| GET від одного користувача багато разів | ❌ НЕ потрібно | Читання не змінює дані |
| POST від одного користувача багато разів | ✅ ПОТРІБНО | Спам, DDoS БД |
| DELETE від одного користувача | ✅ КРИТИЧНО ПОТРІБНО | Необоротне видалення |
| API запити від ботів | ✅ ОБОВ'ЯЗКОВО | Захист від вирусів |
| Масові операції (import CSV) | ⚠️ ЗАЛЕЖИТЬ | Можна дозволити більший ліміт |
| Адміністративні операції | ✅ КРИТИЧНО ПОТРІБНО | Гарантія дозволу конкретним користувачам |

## Висновок

Вибір POST методів правильний, бо:
- ✅ Операції, що змінюють дані, більш уразливі
- ✅ POST менш частовий, тому більш обґрунтований ліміт
- ✅ DeleteConfirmed найкритичніше, має найменший ліміт
- ✅ Відповідає бізнес-логіці захисту даних
